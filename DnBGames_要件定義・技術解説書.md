# DnB Games 要件定義・技術解説書

作成日: 2026-06-11 / 対象バージョン: 0.1.0

---

## 1. プロジェクト概要

### 1.1 背景と目的

Dual N-Back（DnB）は、N回前に提示された刺激と現在の刺激が一致するかを判断し続けることでワーキングメモリーに負荷をかけるトレーニング課題である。Jaeggiら（2008）の研究で流動性知能への転移効果が報告されて以来、Brain Workshopをはじめとする実装が広く使われてきた。

本プロジェクトの目的は、クラシックなDual N-Backを核としつつ、複数のバリエーションモードを単一のゲームエンジンで動かせるブラウザゲームを構築することである。設計上の最重要目標は「**新しいモードの追加コストを最小化する**」ことに置いた。

### 1.2 スコープ

本バージョンに含むもの：8つのゲームモード、キーボードと画面ボタンの両対応、Web Speech APIによる音声刺激、localStorageによる履歴・最高記録・前回設定の永続化、成績に応じた難易度（N）の自動調整。

本バージョンに含まないもの：サーバーサイドのデータ保存、ユーザーアカウント、ランキング、モバイルアプリ化。これらは将来の拡張候補として第10章で触れる。

## 2. 機能要件

### 2.1 ゲームモード一覧

| ID | モード名 | 刺激チャンネル | 判定ルール | 既定N |
|---|---|---|---|---|
| `classic` | クラシック Dual N-Back | 位置＋音 | N回前と一致 | 2 |
| `single-position` | シングル（位置） | 位置 | N回前と一致 | 2 |
| `single-audio` | シングル（音） | 音 | N回前と一致 | 2 |
| `triple` | Triple N-Back | 位置＋音＋色 | N回前と一致 | 2 |
| `variable` | Variable N-Back | 位置＋音 | 一致（N自動増減） | 2 |
| `arithmetic` | Arithmetic N-Back | 数字 | N回前との合計が偶数 | 1 |
| `pattern` | Pattern N-Back | 図形 | N回前と一致 | 2 |
| `stroop` | Color-Word N-Back | 色文字 | 文字の「意味」が一致 | 2 |

### 2.2 セッション仕様

1セッションの試行数は `20 + N`（Brain Workshop準拠）。各試行は「刺激提示（提示間隔の30%、最大700ms）→ 消灯 → インターバル満了で次試行」のサイクルで進行する。提示間隔はゆっくり（3000ms）・標準（2500ms）・速い（2000ms）から選択できる。

最初のN試行は比較対象が存在しないため応答を受け付けず、UIにも「記憶するだけ」と明示する。応答は試行ごとにチャンネル単位で1回のみ受け付け、押した瞬間に正誤フィードバック（緑／赤）を返す。

### 2.3 応答方法

画面ボタンとキーボードの両方に対応する。キー割り当てはチャンネル定義に持たせており、位置=`A`、音=`L`、色=`F`。単一チャンネルのモードはそのチャンネルのキーをそのまま使う。キーリピート（押しっぱなし）は無視する。

### 2.4 マッチ生成仕様

完全ランダムな刺激列ではマッチ率が低すぎる（位置なら約1/9）ため、トレーニングとして成立しない。そこで各チャンネルに注入率（matchRate）を持たせ、「N回前と一致する刺激」を意図的に混入させる。

| チャンネル | 値の種類数 | 注入率 | 実測マッチ率（偶然込み） |
|---|---|---|---|
| 位置 | 9 | 25% | 約34% |
| 音 | 8 | 25% | 約34% |
| 色 | 6 | 25% | 約37% |
| 図形 | 6 | 25% | 約38% |
| ことば（ストループ） | 4 | 15% | 約36% |
| 数字（算術） | 9 | 0%（注入なし） | 約50% |

実測マッチ率は `注入率 + (1 − 注入率) × 偶然一致率` で説明でき、値の種類が少ないチャンネルほど偶然一致が増える。このため図形・ことばは注入率を下げて全チャンネルの体感難度を揃えている。算術モードは「合計の偶奇」という判定の性質上、自然発生率が約50%になるため注入しない。

### 2.5 採点仕様（信号検出理論ベース）

判定対象は `trialIndex >= N` の試行のみ。各試行×各チャンネルを以下の4分類で集計する。

| | 応答した | 応答しなかった |
|---|---|---|
| **マッチだった** | hit（的中） | miss（見逃し） |
| **マッチでなかった** | falseAlarm（誤反応） | correctRejection（正棄却） |

チャンネル正答率 = `(hits + correctRejections) / 判定対象試行数`。総合正答率は全チャンネルの単純平均。「押さなければ減点されない」単純な的中率ではなく誤反応もペナルティになるため、常時連打する戦略は成立しない。

### 2.6 適応アルゴリズム（Variable N-Back）

Jaeggiプロトコルに倣い、セッション間でNを調整する。

```
総合正答率 ≥ 80%  →  N + 1（上限9）
総合正答率 < 50%  →  N − 1（下限1）
それ以外          →  据え置き
```

セッション中のN変更は行わない（試行間の比較整合性が崩れるため）。調整後のNはlocalStorageに保存され、次回プレイ時に自動で設定される。

### 2.7 データ永続化

localStorageに以下を保存する。すべて `dnb:` プレフィックスで名前空間を分離している。

| キー | 内容 |
|---|---|
| `dnb:history` | セッション結果の配列（最新50件） |
| `dnb:best:<modeId>` | モードごとの最高記録（N優先、同Nなら正答率で比較） |
| `dnb:lastN:<modeId>` | モードごとの前回N（adaptiveは推奨N） |

読み書きはすべてtry-catchでラップし、容量超過や無効JSONでもゲーム進行を止めない。SSR環境では `typeof window` ガードで安全に無効化される。

## 3. 非機能要件

**アクセシビリティ**: `:focus-visible` による可視フォーカスリング、`prefers-reduced-motion` の尊重（アニメーション無効化）、刺激表示への `role="img"` と `aria-label` 付与。

**堅牢性**: 音声合成・localStorage・タイマーの失敗がゲーム本体を停止させないこと。React Strict Modeの二重実行下でもセッション結果が二重保存されないこと。

**保守性**: ゲームロジック（シーケンス生成・採点・適応）はReact非依存の純粋TypeScriptとし、単体でテスト可能であること。

**対応環境**: モダンブラウザ（Chrome / Edge推奨）。音声刺激はWeb Speech API対応ブラウザのみ。モバイル幅（約380px）まで崩れないレスポンシブ対応。

## 4. 技術スタック選定

| 技術 | バージョン | 選定理由 |
|---|---|---|
| Next.js (App Router) | 14.2.35 | 安定性重視。React 18と組み合わせ可能な最後のメジャー系列で、既存プロジェクト（library-system）と運用知見を共有できる |
| React | 18.3.1 | Next.js 14の対応範囲。react-domと完全に同一バージョンに固定 |
| TypeScript | 5系 | チャンネル抽象化（判定関数の型）に静的型が不可欠 |
| Tailwind CSS | 3.4系 | デザイントークンを`tailwind.config.ts`に集約。4系はNext 14との組み合わせ実績が浅いため見送り |
| Web Speech API | — | 音声ファイル不要で8文字の読み上げを実現。ビルドサイズゼロ |

### 4.1 依存関係の固定方針（重要）

`next` と `eslint-config-next` は**必ず同一バージョン**（14.2.35）に固定している。これは別プロジェクトで実際に発生した障害——`next@16`系と`eslint-config-next@14`系の混在によりESLintのpeer dependency競合（ERESOLVE）が起き、`node_modules`が不整合状態になった——の再発防止策である。どちらかを上げる場合は必ずセットで上げること。

### 4.2 フォント戦略

Google Fontsを`<link>`タグで読み込む（Space Grotesk＝表示用、Zen Kaku Gothic New＝日本語本文、IBM Plex Mono＝数値）。`next/font/google`を使わない理由は、ビルド時のフォント取得失敗がビルド自体を壊すリスクを避けるため。`<link>`方式なら取得失敗時もフォールバックフォントで動作する。

## 5. アーキテクチャ

### 5.1 設計の核心：チャンネル抽象化

本プロジェクトの最重要設計判断。8つのモードは一見バラバラだが、すべて次のループに還元できる。

```
シーケンス生成 → 刺激提示 → 入力受付 → N-back判定 → スコア集計
```

モード間の差分は「刺激チャンネルの組み合わせ」「判定ルール」「Nの制御方法」の3点のみ。そこで刺激を `ChannelDef` として抽象化した。

```typescript
interface ChannelDef {
  id: ChannelId;
  label: string;       // UI表示名
  key: string;         // 応答キー
  matchRate: number;   // マッチ注入率（0なら自然発生に任せる）
  random: (rng) => string;             // ランダム刺激の生成
  makeMatch: (prev, rng) => string;    // N回前にマッチする刺激の生成
  isMatch: (cur, prev) => boolean;     // マッチ判定
  matchLabel: string;  // ボタン文言
}
```

`random` / `makeMatch` / `isMatch` の3関数を差し替えるだけで、判定ロジックの異なるモードを表現できる。

```typescript
// 通常: 単純一致
isMatch: (cur, prev) => cur === prev

// 算術: 合計の偶奇
isMatch: (cur, prev) => (Number(cur) + Number(prev)) % 2 === 0

// ストループ: 値は "単語ID:インクID" 形式。意味（単語部分）のみで判定
isMatch: (cur, prev) => stroopWordOf(cur) === stroopWordOf(prev)
// makeMatch は単語を保持しインクだけランダム化 → 干渉を維持したままマッチを注入できる
```

モードは `GameMode`（チャンネル配列＋既定N＋表示方式＋adaptiveフラグ）という設定オブジェクトであり、`modes.ts` の配列に1件追加するだけで新モードが完成する。エンジン・UI側の変更は一切不要。

`lib/nback/` がReactに一切依存しないことが要点。シミュレーションテスト（後述）はこの層だけをts-nodeで直接実行している。

### 5.3 状態機械（useNBackGame）

ゲーム進行は `useReducer` による状態機械で管理する。

```
idle → countdown(3→2→1) → running(試行0..N+19) → finished
```

設計上のポイントが3つある。

**(1) reducerにロジックを集約し、effectはタイマー編成のみを担う。**
試行の終了判定や採点は `ADVANCE` アクション内でreducerが純粋計算として行う。effect側のクロージャに古い状態（特に応答データ）が残る「stale closure」問題を、状態遷移をreducerに寄せることで構造的に回避している。

**(2) 試行effectの依存配列は `[phase, index]` のみ。**
応答（`RESPOND`）による再レンダリングでタイマーが再起動しないよう、依存を意図的に絞っている。これにより1試行につき提示タイマーと終了タイマーがちょうど1回ずつ動く。

**(3) Strict Mode下の二重保存ガード。**
`reactStrictMode: true` では開発時にeffectが二重実行される。セッション保存effectは結果のISO日時を `useRef` に記録し、同一結果の二重保存を防ぐ。

### 5.4 UIコンポーネント設計

| コンポーネント | 責務 |
|---|---|
| `PlayClient` | フェーズに応じた画面切替（設定／カウントダウン／プレイ／結果）、N・速度の管理 |
| `Grid` | 3×3グリッド。Tripleでは色チャンネルの値でセルを着色 |
| `CenterStimulus` | 数字・図形・色文字・音（♪パルス）の中央大型表示 |
| `ResponseButtons` | チャンネルごとのボタン。キー表記と即時正誤フィードバック |
| `TrialDots` | 進行バー兼成績記録。過去試行を全問正解=緑／誤りあり=赤で塗る |
| `ResultPanel` | チャンネル別の成績表、自己ベスト更新表示、適応Nの予告 |

`TrialDots` は本UIのシグネチャ要素で、「進行表示」と「どの試行でミスしたか」を1つのコンポーネントで兼ねる。

音のみのモードで文字を表示しない（♪のパルスのみ見せる）のは意図的な仕様である。文字を表示すると視覚課題に変質してしまうため、提示タイミングだけを可視化している。

## 7. テスト戦略

ロジック層に対し、1000セッション×全8モードのシミュレーションテスト（`tests/simulate.ts`）を実装・実行済み。検証項目は以下。

1. **シーケンス整合性**: 全試行・全チャンネルで刺激値が有効、最初のN試行はマッチ不可、`isMatch` フラグが判定関数の再計算と一致する
2. **マッチ率**: 各チャンネルの実測マッチ率が `注入率 + 偶然一致` の理論値帯に収まる
3. **採点の境界**: 完璧なプレイヤー（マッチ時のみ応答）で総合100%・miss/falseAlarmゼロ、無応答プレイヤーでhit/falseAlarmゼロかつ判定数の合計が一致
4. **適応則の境界値**: 80%ちょうどでN+1、79%で据え置き、49%でN−1、下限1・上限9
5. **ストループ固有**: `makeMatch` が単語の意味を保持したままインクのみ変える

UI層（タイマー・キーボード・描画）は現状手動確認。将来的にはPlaywrightでのE2E（開始→全試行応答→結果表示）を追加するのが妥当。

## 8. 既知の制約と注意点

**Web Speech APIの自動再生制限**: 一部ブラウザはユーザー操作前の発声をブロックする。本実装は「開始」ボタンのクリックを必ず経由するため通常は問題にならないが、Edge等で音が出ない場合は最初の発声のみスキップされる可能性がある。

**合成音声の品質**: 読み上げ品質と遅延はOS・ブラウザの音声エンジンに依存する。8文字（C/H/K/L/Q/R/S/T）は聞き分けやすさで選定済みだが、環境によっては「C」と「T」が紛らわしい場合がある。

**localStorageの限界**: 同一ブラウザ・同一オリジン内でのみ永続。ブラウザのデータ削除で消える。複数端末間の同期は将来のサーバー保存で対応する領域。

**タイマー精度**: `setTimeout` ベースのため、タブが非アクティブになるとブラウザのスロットリングで提示間隔が乱れる。プレイ中はタブを前面に保つ前提。

## 9. 新モード追加ガイド

手順は2ステップ。エンジン・UIの変更は不要。

**Step 1**: 新しい刺激が必要なら `channels.ts` に `ChannelDef` を追加（`random` / `makeMatch` / `isMatch` の3関数を定義）。

**Step 2**: `modes.ts` の `MODES` 配列にモード定義を1件追加。

```typescript
// 例: 位置＋図形のDualモード
{
  id: "dual-visual",
  name: "Dual（位置＋図形）",
  tagline: "音を使わない視覚Dual",
  group: "バリエーション",
  channels: [positionChannel, shapeChannel],
  defaultN: 2,
  display: "grid",
  baseTrials: 20,
}
```

注意点は2つ。同一モード内でチャンネルの応答キーが重複しないこと（位置とシェイプはどちらも`a`なので、片方のキーを変えた `ChannelDef` を作る）。値の種類が少ないチャンネルは偶然一致が増えるため、注入率を下げて体感難度を調整すること（第2.4節の表を参照）。

## 10. 将来の拡張候補

初期検討時に挙がったが本バージョンでは見送ったモード・機能。

- **Survival / Endlessモード**: ライフ制でどこまで続くか。エンジンの試行数固定を「失敗カウント打ち切り」に拡張すれば実装可能
- **Time Attack**: 提示間隔がセッション中に短縮されていく。`intervalMs` を試行関数化する
- **Daily Challenge**: 日付シードの固定シーケンス。`generateTrials` は既に `rng` を注入可能な設計のため、シード付き乱数（mulberry32等）を渡すだけで対応できる
- **サーバー保存とランキング**: localStorage層と同じインターフェースのリモート実装に差し替え
- **統計グラフ**: 履歴データからNと正答率の推移を可視化
